# Nevin production deployment

Production runs the yopass instance at `yopass.nevin.se`, fronted by
Cloudflare in **Full SSL** mode. The instance terminates TLS itself with a
self-signed cert generated by cloud-init at first boot; Cloudflare validates
that cert (just for "is the connection encrypted at all"), and the inbound
security group only admits Cloudflare's IPv4 ranges, so direct origin access
bypassing CF is blocked.

This differs from `deploy/nevin-sandbox/` in three ways:

1. **No shared Caddy** — production has no reverse-proxy module. Yopass binds
   to `:443` with `--tls-cert /tls/origin.crt --tls-key /tls/origin.key`.
2. **Different registry** — `rg.nl-ams.scw.cloud/nevin-production/yopass`,
   pulled with the `yopass-watchtower` IAM key (read-only) provisioned via
   `nevin-platform-infra/environments/production/`.
3. **Manual deploys only** — see `.github/workflows/deploy-production.yml`,
   triggered via `gh workflow run deploy-production.yml` or the GitHub UI.

## Deployment model

Same Watchtower-pull as sandbox, but each push to production has to be
explicitly dispatched (no auto-deploy on commit):

```
gh workflow run deploy-production.yml      → builds & pushes :latest + :sha
   (manual trigger)                          to nevin-production registry
                                           ↓
                                 Watchtower (≤60s poll)
                                           ↓
                          docker compose recreate yopass
```

End-to-end: ~3 min from `gh workflow run` to live, same as sandbox.

## First-time setup on the production instance

Get the values:

```sh
cd ~/repos/nevin/nevin-platform-infra
make production-output
# → yopass_public_ipv4, yopass_watchtower_application_id (and yopass_ci_application_id for the GH side)
```

Mint the Watchtower API key (read-only):

```sh
scw iam api-key create application-id=<yopass_watchtower_application_id> \
  default-project-id=<production project id>
# Copy the secret_key — only shown once.
```

SSH in and bootstrap:

```sh
ssh deploy@<yopass_public_ipv4>
mkdir -p ~/yopass && cd ~/yopass
# scp this directory's docker-compose.yml here from your laptop:
#   scp deploy/nevin-production/docker-compose.yml deploy@<ip>:~/yopass/

# One-time docker login → seeds ~/.docker/config.json for Watchtower:
echo "<yopass-watchtower secret_key>" | docker login rg.nl-ams.scw.cloud \
  -u nologin --password-stdin

docker compose up -d
```

After this, `docker compose ps` shows three services running: `yopass`,
`memcached`, `watchtower`.

## GitHub Actions setup (one-time)

Mint the CI key (full registry access):

```sh
scw iam api-key create application-id=<yopass_ci_application_id> \
  default-project-id=<production project id>
```

In `nevintech/yopass` GitHub repo settings:

1. Settings → Environments → **New environment** named `production`.
2. (Optional) Add required reviewers / wait timers / branch restrictions
   under the environment's protection rules.
3. Environment secrets → add `SCW_SECRET_KEY` with the `yopass-ci` secret
   key from the previous step.

That's it — no `DEPLOY_HOST`/`DEPLOY_USER`/`DEPLOY_SSH_KEY` needed; CI's
job is just to build and push.

## Rolling out a release

```sh
gh workflow run deploy-production.yml -R nevintech/yopass
gh run watch -R nevintech/yopass    # optional — tail the build
```

Or via the GitHub UI: Actions → "Build and push (production)" → Run workflow.

If you've configured a required reviewer on the environment, the workflow
will pause for approval after the build step (or at the start, depending on
your GitHub settings).

## Verifying

```sh
curl -fsS https://yopass.nevin.se/api/health         # 200 OK
curl -s   https://yopass.nevin.se/config | jq .APP_NAME
```

Then create a one-time secret in the browser, open the link once, reload —
should 404 on the second visit.

## Updating the on-instance compose

Same caveat as sandbox: Watchtower auto-updates the **image**, but changes
to `docker-compose.yml` (flags, volumes, watchtower config) need a manual
scp + recreate:

```sh
scp deploy/nevin-production/docker-compose.yml deploy@<ip>:~/yopass/
ssh deploy@<ip> 'cd ~/yopass && docker compose up -d --force-recreate'
```
